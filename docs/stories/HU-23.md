# HU-23 — Actualización del Diccionario de Ejercicios y Plan de Entrenamiento por defecto

## Requisitos relacionados

RF04, RF05 (modificación)

## Descripción

Como ejecutante, necesito que al instalar la aplicación por primera vez o al actualizar desde una versión anterior, el diccionario de ejercicios precargado y el plan de entrenamiento por defecto reflejen exactamente el catálogo de 26 ejercicios y las 3 rutinas (Pierna con 2 versiones, Push con 1 versión, Pull con 1 versión) definidos en la documentación vigente, para comenzar a entrenar inmediatamente con un plan actualizado y coherente, sin perder la capacidad de crear ejercicios propios ni editar los de base.

El diccionario pasa de 43 ejercicios precargados a 26. Se eliminan 22 ejercicios obsoletos, se renombran varios existentes, se ajustan tipos de equipamiento y zonas musculares, y se agregan 14 ejercicios nuevos. El plan de entrenamiento por defecto pasa de 3 rutinas × 3 versiones cada una (9 versiones, 82 asignaciones) a 3 rutinas con 4 versiones totales (Pierna V1+V2, Push V1, Pull V1) y 27 asignaciones. Se introducen 6 nuevos tipos de equipamiento para reflejar la variabilidad de implementos que admiten ciertos ejercicios.

### Decisiones de Diseño

**Patrón Arquitectónico:** Migración DDL + Data (MIGRATION_9_10) + Rewrite completo de Seeders + Asset cleanup. No introduce patrón arquitectónico nuevo — la misma separación de 4 capas MVVM + Domain Layer (ADR-05) se mantiene intacta. Los cambios se concentran exclusivamente en la capa Data: migración SQL/Kotlin, seeders de datos precargados y assets de imágenes.

**Justificación:** La HU-23 es una historia de **datos puros** — no cambia lógica de negocio, UI (salvo 1 método de carga de imágenes), navegación ni reglas de dominio. El esquema de tablas no cambia (las 18 entidades de v9 se preservan con exactamente las mismas columnas y constraints). Lo que cambia es: (1) el **catálogo de datos** — 6 equipment types nuevos, 43 ejercicios reemplazados por 26, plan reestructurado de 9 versiones/82 asignaciones a 4 versiones/27 asignaciones; (2) la **migración para usuarios existentes** — lógica compleja con condiciones de borde (historial, sesión activa, deload activo, ejercicios custom, versiones con sesiones históricas); (3) los **seeders para nuevas instalaciones** — `BaseDataSeeder`, `ExerciseSeeder`, `PlanSeeder` se reescriben completamente; (4) los **assets** — 29 imágenes a eliminar, 14 a renombrar, 12 nuevas ya provistas. Se sigue el mismo patrón de migración establecido por MIGRATION_6_7 (HU-15), MIGRATION_7_8 (HU-20) y MIGRATION_8_9 (HU-22), donde la migración se implementa como `object : Migration(N, N+1)` dentro de `Migrations.kt` y se registra en `DatabaseModule`.

**Bloque A — MIGRATION_9_10 en `Migrations.kt` (Nuevo)**

- **`MIGRATION_9_10` (Nuevo):** Migración SQL + Kotlin que actualiza catálogo de ejercicios, tipos de equipamiento y plan de entrenamiento sin pérdida de datos del usuario. Implementada en 6 fases secuenciales dentro del método `migrate()`:
  - Nivel de cambio: Crítico
  - Ubicación: `data/local/database/Migrations.kt`
  - Especificaciones:

    **Fase 1 — Equipment types nuevos (CA-14):**

    ```sql
    INSERT INTO equipment_type (id, name) VALUES (10, 'Mancuernas o Polea');
    INSERT INTO equipment_type (id, name) VALUES (11, 'Polea con Cuerda');
    INSERT INTO equipment_type (id, name) VALUES (12, 'Polea con Cuerda o Polea con Barra en V');
    INSERT INTO equipment_type (id, name) VALUES (13, 'Barra');
    INSERT INTO equipment_type (id, name) VALUES (14, 'Mancuerna o Polea o Barra');
    INSERT INTO equipment_type (id, name) VALUES (15, 'Barra o Mancuernas');
    ```

    **Fase 2 — Completar deload activo si existe (CA-28):**

    ```sql
    UPDATE deload SET status = 'COMPLETED', completion_date = date('now')
    WHERE status = 'ACTIVE';
    ```

    Esto se ejecuta ANTES de tocar rutinas/versiones para evitar inconsistencias en `deload_frozen_version` que referencia `routine_id`.

    **Fase 3 — Cerrar sesión activa si referencia versión que será eliminada (CA-22):**

    ```sql
    UPDATE session SET status = 'COMPLETED'
    WHERE status = 'IN_PROGRESS'
      AND routine_version_id IN (
        SELECT id FROM routine_version WHERE routine_id IN (1, 2, 3)
      );
    ```

    Si la sesión activa pertenece a una rutina del usuario (ID > 3), no se toca. Si pertenece a una rutina seed (1, 2, 3) cuya versión será eliminada, se cierra automáticamente.

    **Fase 4 — Limpieza de plan y rutinas originales (CA-17, CA-18, CA-19, CA-27):**

    ```sql
    -- Eliminar asignaciones de versiones de rutinas seed
    DELETE FROM plan_assignment
    WHERE routine_version_id IN (
      SELECT id FROM routine_version WHERE routine_id IN (1, 2, 3)
    );

    -- Eliminar deload_frozen_version de rutinas seed
    DELETE FROM deload_frozen_version WHERE routine_id IN (1, 2, 3);

    -- Eliminar routine_current_version de rutinas seed
    DELETE FROM routine_current_version WHERE routine_id IN (1, 2, 3);

    -- Eliminar alertas de rutinas seed (CA-27)
    DELETE FROM alert WHERE routine_id IN (1, 2, 3);

    -- Eliminar routine_versions que NO tienen sesiones asociadas
    DELETE FROM routine_version
    WHERE routine_id IN (1, 2, 3)
      AND id NOT IN (SELECT DISTINCT routine_version_id FROM session);

    -- Eliminar rutinas seed SOLO si no retienen versiones por historial
    DELETE FROM routine WHERE id IN (1, 2, 3)
      AND id NOT IN (SELECT DISTINCT routine_id FROM routine_version);
    ```

    **Diseño clave — Preservación de versiones con historial (CA-25):** Si una `routine_version` con `routine_id IN (1, 2, 3)` tiene sesiones históricas, NO se elimina. La rutina padre tampoco se elimina (FK constraint). Las nuevas rutinas se crean con IDs dinámicos (MAX+1), evitando colisiones. Las rutinas legacy quedan como "huérfanas" — sin `routine_current_version`, sin `plan_assignment`, sin alertas — pero sus versiones preservan la integridad referencial de `session.routine_version_id`. Esto es correcto: el usuario puede ver el historial de sesiones pasadas con sus versiones originales.

    **Fase 5 — Ejercicios: eliminar obsoletos, preservar con historial (CA-15, CA-23, CA-26):**

    ```sql
    -- Marcar como custom los ejercicios precargados que tienen historial
    -- (incluye tanto exercise_id como original_exercise_id para cubrir
    -- ejercicios que fueron sustituidos en sesión — FK RESTRICT en ambos)
    UPDATE exercise SET is_custom = 1
    WHERE is_custom = 0
      AND id IN (
        SELECT DISTINCT se.exercise_id FROM session_exercise se
        UNION
        SELECT DISTINCT se.original_exercise_id FROM session_exercise se
        WHERE se.original_exercise_id IS NOT NULL
      );

    -- Eliminar alertas vinculadas a ejercicios sin historial (CA-27)
    -- (FK RESTRICT en alert.exercise_id → exercise.id impide eliminar
    -- ejercicios que tengan alertas sin limpiarlas primero)
    DELETE FROM alert
    WHERE exercise_id IS NOT NULL
      AND exercise_id IN (
        SELECT id FROM exercise
        WHERE is_custom = 0
          AND id NOT IN (SELECT DISTINCT exercise_id FROM session_exercise)
          AND id NOT IN (
            SELECT DISTINCT original_exercise_id FROM session_exercise
            WHERE original_exercise_id IS NOT NULL
          )
      );

    -- Eliminar exercise_progression de ejercicios sin historial (CA-26)
    DELETE FROM exercise_progression
    WHERE exercise_id IN (
      SELECT id FROM exercise
      WHERE is_custom = 0
        AND id NOT IN (SELECT DISTINCT exercise_id FROM session_exercise)
        AND id NOT IN (
          SELECT DISTINCT original_exercise_id FROM session_exercise
          WHERE original_exercise_id IS NOT NULL
        )
    );

    -- Eliminar exercise_muscle_zone de ejercicios sin historial
    DELETE FROM exercise_muscle_zone
    WHERE exercise_id IN (
      SELECT id FROM exercise
      WHERE is_custom = 0
        AND id NOT IN (SELECT DISTINCT exercise_id FROM session_exercise)
        AND id NOT IN (
          SELECT DISTINCT original_exercise_id FROM session_exercise
          WHERE original_exercise_id IS NOT NULL
        )
    );

    -- Eliminar plan_assignment de ejercicios sin historial
    -- (versiones seed ya limpiadas en Fase 4; esto cubre asignaciones
    -- en rutinas del usuario que referencien ejercicios precargados)
    DELETE FROM plan_assignment
    WHERE exercise_id IN (
      SELECT id FROM exercise
      WHERE is_custom = 0
        AND id NOT IN (SELECT DISTINCT exercise_id FROM session_exercise)
        AND id NOT IN (
          SELECT DISTINCT original_exercise_id FROM session_exercise
          WHERE original_exercise_id IS NOT NULL
        )
    );

    -- Eliminar ejercicios precargados sin historial
    DELETE FROM exercise
    WHERE is_custom = 0
      AND id NOT IN (SELECT DISTINCT exercise_id FROM session_exercise)
      AND id NOT IN (
        SELECT DISTINCT original_exercise_id FROM session_exercise
        WHERE original_exercise_id IS NOT NULL
      );
    ```

    **Nota:** Todos los ejercicios precargados (IDs 1-43 originales) que no tienen sesiones ni referencias como `original_exercise_id` se eliminan. Los que tienen sesiones (directas o como ejercicio sustituido) se marcan `is_custom = 1` y permanecen en el catálogo del usuario con sus zonas musculares y progresión intactas. Los ejercicios custom del usuario (`is_custom = 1` previo) no se tocan (CA-23). Las alertas vinculadas a ejercicios eliminados se limpian explícitamente antes del DELETE para respetar la FK RESTRICT en `alert.exercise_id`.

    **Fase 6 — Insertar nuevos ejercicios + plan (CA-03, CA-08-CA-11, CA-16, CA-20, CA-21):**

    Esta fase se implementa en **Kotlin** (no SQL puro) dentro del método `migrate()` porque los IDs son dinámicos (MAX+1):

    ```kotlin
    // Obtener MAX exercise ID para evitar colisiones (CA-16)
    val maxExerciseId = db.query("SELECT COALESCE(MAX(id), 0) FROM exercise")
        .use { if (it.moveToFirst()) it.getLong(0) else 0L }
    val baseExId = maxExerciseId + 1

    // Insertar 26 ejercicios con IDs baseExId..baseExId+25
    // Insertar 29 registros en exercise_muscle_zone (26 ejercicios, 3 con doble zona)

    // Obtener MAX routine ID para nuevas rutinas
    val maxRoutineId = db.query("SELECT COALESCE(MAX(id), 0) FROM routine")
        .use { if (it.moveToFirst()) it.getLong(0) else 0L }
    val piernaId = maxRoutineId + 1
    val pushId = maxRoutineId + 2
    val pullId = maxRoutineId + 3

    // Insertar 3 rutinas (CA-08 adaptado a IDs dinámicos)
    // Insertar 4 routine_versions con IDs MAX+1..+4 (CA-09)
    // Insertar 3 routine_current_version con version_number = 1 (CA-20)
    // Insertar 27 plan_assignments (CA-11 con exercise IDs mapeados)

    // Reiniciar rotación (CA-21)
    db.execSQL("UPDATE rotation_state SET microcycle_position = 1 WHERE id = 1")
    ```

    **Mapeo de IDs lógicos a dinámicos:** Los ejercicios se definen en CA-03 con IDs lógicos 1-26. En migración, se insertan con IDs `baseExId + (idLógico - 1)`. Las asignaciones del plan (CA-11) referencian IDs lógicos que deben traducirse a IDs reales usando este mismo offset. Ejemplo: CA-11 dice `exercise_id = 1` (Aductores) → en migración es `baseExId + 0`.

**Verificaciones críticas:**

- La tabla `exercise` tiene FK `equipment_type_id → equipment_type(id)` con `ON DELETE RESTRICT`. La Fase 1 de migración inserta los 6 nuevos equipment types ANTES de que la Fase 6 inserte ejercicios que los referencien. Orden correcto verificado.
- La tabla `exercise_muscle_zone` tiene FK `exercise_id → exercise(id)` con `ON DELETE RESTRICT`. La Fase 5 elimina registros de `exercise_muscle_zone` ANTES de eliminar el ejercicio padre. Orden correcto verificado.
- La tabla `plan_assignment` tiene FK `exercise_id → exercise(id)` con `ON DELETE RESTRICT`. La Fase 4 elimina asignaciones de versiones seed. La Fase 5 elimina asignaciones restantes de ejercicios sin historial. Ambas ANTES de eliminar ejercicios. Orden correcto verificado.
- La tabla `session_exercise` tiene FK `exercise_id → exercise(id)` con `ON DELETE RESTRICT` y FK `original_exercise_id → exercise(id)` con `ON DELETE RESTRICT`. Los ejercicios referenciados por cualquiera de las dos columnas NO se eliminan (se marcan `is_custom = 1`). La integridad de `session_exercise` se preserva.
- La tabla `alert` tiene FK `exercise_id → exercise(id)` con `ON DELETE RESTRICT`. Las alertas de ejercicios sin historial se eliminan explícitamente en Fase 5 ANTES de eliminar el ejercicio padre.
- La tabla `exercise_progression` tiene FK `exercise_id → exercise(id)`. La Fase 5 elimina progresiones de ejercicios sin historial ANTES de eliminar el ejercicio. Las progresiones de ejercicios preservados (marcados custom) se mantienen (CA-26).
- `RotationState` singleton (id=1) — la Fase 6 reinicia `microcycle_position = 1` pero preserva `microcycle_count` (CA-21).
- Las migraciones Room son transaccionales por defecto — si MIGRATION_9_10 falla a mitad, se revierte completamente.

| Rutina seed (ID 1/2/3) tiene versiones con sesiones históricas → no se puede eliminar | FK violation si se intenta DELETE de rutina | Fase 4 elimina solo `routine_version` sin sesiones. Si quedan versiones, la rutina se preserva. Nuevas rutinas se crean con IDs MAX+1 |
| Sesión activa al momento de migración | Sesión queda con FK a versión eliminada | Fase 3 cierra sesiones activas en versiones de rutinas seed ANTES de que Fase 4 las elimine (CA-22) |
| Deload activo referencia rutinas eliminadas | FK violation en `deload_frozen_version` | Fase 2 completa deload activo ANTES de tocar rutinas (CA-28). Fase 4 elimina `deload_frozen_version` de rutinas seed |
| IDs de ejercicios nuevos colisionan con ejercicios custom del usuario | UNIQUE constraint violation en `exercise(name, equipment_type_id)` | CA-16: IDs nuevos parten de MAX(id)+1, garantiza no colisión de PKs. El UNIQUE de (name, equipment_type_id) también está protegido porque los nombres son nuevos y no duplican nombres existentes |
| Ejercicio precargado referenciado por `original_exercise_id` (sustitución en sesión) se intenta eliminar | FK RESTRICT violation en `session_exercise.original_exercise_id` | Fase 5 verifica TANTO `exercise_id` COMO `original_exercise_id` en `session_exercise` antes de marcar/eliminar. Ejercicios referenciados por sustitución se preservan como custom |
| Alerta referencia ejercicio precargado sin historial que se va a eliminar | FK RESTRICT violation en `alert.exercise_id` | Fase 5 incluye `DELETE FROM alert WHERE exercise_id IN (...)` explícito ANTES de eliminar ejercicios. Alertas de ejercicios preservados (custom) se mantienen |
| Imágenes con nombre incorrecto en assets | Ejercicio muestra placeholder en vez de imagen | Validar que los 26 `media_resource` mapean 1:1 a archivos existentes en `assets/exercises/` después de aplicar renames y deletes |
| `rememberExerciseBitmap()` busca en subdirectorios que ya no existen | Imágenes NO se muestran para ejercicios precargados en versión actual (v9) | La corrección en Bloque E es **retrocompatible**: si la app v9 actual tiene este bug, v10 lo corrige. Los ejercicios custom (ruta absoluta) no se afectan |
| Rutinas del usuario (ID > 3) tienen ejercicios precargados asignados que serán eliminados | plan_assignment con FK rota | Fase 5 elimina `plan_assignment` de ejercicios sin historial que se van a eliminar, independientemente de la rutina. El usuario pierde esas asignaciones pero preserva su rutina y versiones |

### Notas Técnicas

- **Scope de la migración:** MIGRATION_9_10 es más simple que MIGRATION_8_9 (HU-22) — no cambia esquema de tablas, solo manipula datos. Sin embargo, la lógica condicional (preservar ejercicios con historial, IDs dinámicos, versiones con sesiones) es más compleja que las migraciones anteriores. Se recomienda test de migración instrumentado con `MigrationTestHelper` usando escenarios: (1) usuario limpio sin datos, (2) usuario con historial en ejercicios 8, 10, 16, (3) usuario con sesión activa en versión 4, (4) usuario con deload activo, (5) usuario con ejercicios custom y rutinas propias.
- **IDs dinámicos en migración vs IDs fijos en seeders:** En `ExerciseSeeder` (nuevas instalaciones) los IDs son fijos 1-26 (no hay datos previos). En MIGRATION_9_10, los IDs parten de MAX+1 (puede haber ejercicios custom con IDs > 43). Las `plan_assignment` de la migración deben mapear IDs lógicos (1-26) a IDs reales (baseExId+offset). Los seeders NO necesitan este mapeo.
- **Ejercicios que cambian pero comparten nombre raíz:** Algunos ejercicios del diccionario nuevo tienen nombres similares a los anteriores (ej: "Extensión de Cuádriceps" ID 30 → "Extensión de Cuádriceps" nuevo ID 11). Si el ejercicio viejo fue marcado `is_custom = 1` (por historial), ambos coexisten porque tienen IDs distintos. El UNIQUE (name, equipment_type_id) podría colisionar si el ejercicio viejo tiene el mismo `equipment_type_id`. Verificar: ejercicio viejo ID 30 "Extensión de Cuádriceps" tiene `equipment_type_id = 1` (Máquina) — el nuevo ID 11 también tiene `equipment_type_id = 1`. **Colisión potencial** si el viejo fue marcado custom. Mitigación: en Fase 5, al marcar como custom, renombrar si el nombre colisiona con el nuevo diccionario: `UPDATE exercise SET name = name || ' (anterior)' WHERE ...`.
- **`remo_t_inclinado.png` y `sentadilla_bulgara.png`:** Estos archivos existen en `assets/exercises/` pero CA-07f espera `remo_t_inclinado_maquina.png` y `sentadilla_bulgara_mancuernas.png`. El Developer debe determinar si los archivos actuales son las imágenes correctas que solo necesitan rename, o si son archivos temporales que deben ser reemplazados por las imágenes finales.
- **`PrepopulateFacade` y `PrepopulateCallback`:** No requieren modificación. La fachada invoca los 3 seeders actualizados automáticamente en `onCreate` y `onDestructiveMigration` (CA-30, CA-31).
- **Orden de fases en migración es crítico:** La secuencia Fase 1→2→3→4→5→6 respeta todas las dependencias de FK. Alterar el orden puede causar FK violations (ej: insertar ejercicios con equipment_type_id=10 antes de insertar el equipment_type).
