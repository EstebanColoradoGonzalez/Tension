# HU-25 — Alternativas por puesto en el plan de entrenamiento

## Requisitos relacionados

RF65

## Descripción

Como ejecutante, necesito poder definir ejercicios alternativos para un mismo puesto dentro de mi plan de entrenamiento, de forma que al iniciar una sesión pueda intercambiar cuál de ellos realizar en ese momento. Esto me permite mantener variedad controlada en mi entrenamiento sin tener que modificar el plan manualmente, y refleja la práctica habitual de prescribir opciones equivalentes (e.g., "Hip Thrust ó Sentadilla Búlgara ó Sentadilla de Zumo") para un mismo slot de trabajo.

## Criterios de Aceptación

### Definición de alternativas en el plan

- CA-01: En la vista de detalle de una versión de rutina (D4), cada ejercicio asignado muestra un botón "+" que permite agregar una alternativa para ese mismo puesto (slot).
- CA-02: Al pulsar "+", se abre un bottom sheet donde el usuario puede seleccionar un ejercicio del catálogo para agregarlo como alternativa. El ejercicio seleccionado heredará automáticamente el número de series y repeticiones del ejercicio principal del slot.
- CA-03: Los ejercicios que comparten el mismo `slot` se muestran en la vista de plan como **una sola fila** cuyo titular concatena los nombres con "ó" (e.g., "Hip Thrust ó Sentadilla Búlgara ó Sentadilla de Zumo"). La lista del plan muestra una fila por slot.
- CA-04: Cada fila de slot en el plan tiene su propio botón de agregar alternativa (+), edición de series/repeticiones (✏️) y eliminación (🗑️). La edición de series/repeticiones se propaga automáticamente a todas las alternativas del slot.
- CA-05: Al eliminar una fila de slot, se eliminan **todos** los ejercicios de ese slot (principal y alternativas).
- CA-06: Al eliminar el único ejercicio de un slot (sin alternativas), el comportamiento es el mismo que el existente.
- CA-07: No se permite agregar el mismo ejercicio (mismo `exercise_id`) dos veces en la misma versión de rutina, independientemente del slot.

### Intercambio de ejercicio durante la sesión

- CA-08: Al iniciar una sesión, cada slot crea un `session_exercise` con el ejercicio primario (el primero por `sort_order`). Todos los ejercicios quedan con `exercise_id` asignado y `pending_selection = 0`.
- CA-09: Los slots con un único ejercicio asignado se comportan exactamente igual que antes.
- CA-10: En la pantalla de sesión activa (E1), los ejercicios con alternativas en su slot (determinado por `alternativesInSlot > 1`) muestran un icono de intercambio (↔ SwapHoriz) junto a los botones de "Registrar serie" y "Sustituir". Este icono solo aparece mientras el ejercicio está en estado NOT_STARTED y no tiene series registradas (`completedSets == 0`).
- CA-11: Al pulsar el icono de intercambio, se abre un bottom sheet que lista las alternativas del slot (nombre, equipamiento, resumen de zonas musculares). El usuario selecciona una y confirma.
- CA-12: Al confirmar la selección, el `session_exercise` se actualiza: `exercise_id` se cambia al ejercicio elegido y `original_exercise_id` se limpia a `NULL` (ya que es un cambio entre ejercicios del plan, no una sustitución). La fila se actualiza con el nuevo nombre y datos del ejercicio.
- CA-13: El intercambio de alternativa solo está disponible mientras el ejercicio está NOT_STARTED (sin series registradas). Una vez que el usuario registra al menos una serie, el icono de intercambio desaparece.
- CA-14: El cierre de sesión no se bloquea por ejercicios con alternativas disponibles. El usuario puede cerrar la sesión en cualquier momento.

### Integridad de datos y migración

- CA-15: La base de datos se migra a la versión 13. La migración 11→12 agrega la columna `slot` en `plan_assignment` y las columnas `pending_selection` y `slot` en `session_exercise`. La migración 12→13 corrige los valores de `slot` en `plan_assignment` y `session_exercise` para instalaciones frescas donde el seeder no estableció el slot correctamente.
- CA-16: Los datos existentes en `plan_assignment` se migran automáticamente: `slot` se inicializa con el valor de `sort_order` de cada registro, preservando el orden actual.
- CA-17: Los datos existentes en `session_exercise` se migran con `pending_selection = 0` y con el `slot` correspondiente al ejercicio del plan de esa sesión.

### Restricciones de concurrencia

- CA-18: No se permite agregar alternativas, asignar ejercicios ni eliminar slots mientras haya una sesión activa (IN_PROGRESS) de esa versión de rutina. Los use cases `AddAlternativeToSlotUseCase`, `AssignExerciseToVersionUseCase` y `UnassignExerciseFromVersionUseCase` validan `hasActiveSessionForVersion(routineVersionId)` antes de ejecutar. Si hay sesión activa, se muestra un mensaje al usuario via Snackbar.

## Notas Técnicas

- El `slot` en `plan_assignment` es un entero que agrupa ejercicios del mismo puesto. No es un PK ni un campo único. Múltiples ejercicios pueden compartir el mismo `slot` dentro de una misma `routine_version_id`.
- La clave primaria de `plan_assignment` sigue siendo `(routine_version_id, exercise_id)`. La unicidad de ejercicio por versión se mantiene.
- El índice único de `session_exercise` es `(session_id, exercise_id)`. Dado que un mismo ejercicio solo puede pertenecer a un slot por versión, no puede haber colisiones al intercambiar alternativas.
- Los use cases relacionados son: `AddAlternativeToSlotUseCase` para agregar alternativas al plan. El intercambio en sesión se realiza directamente vía `SessionRepository.switchAlternativeInSession()`.
- `switchAlternativeExercise` en el DAO limpia `original_exercise_id = NULL` al cambiar de alternativa, para que el ejercicio no sea tratado como sustitución sino como un cambio dentro del plan.
- La pantalla de progresión y el cálculo de KPIs solo consideran `session_exercise` con `exercise_id IS NOT NULL` y series registradas.
